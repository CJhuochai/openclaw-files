# Go 学习进度

> 这个文件现在承担两件事：
> 1. 定义**进度口径**，避免“看起来在推进，实际不可核对”
> 2. 记录**当前可信状态**，把学习完成情况和自动提醒情况分开

## 进度口径

### 1. 学习进度（人是否真的完成）
只有满足下面任一条件，某一天才算“已完成”：
- 学习者明确反馈 `Day N done`
- 在当天记录里补齐了复盘/示例/完成勾选
- 有明确人工确认，说明该日已学习完成

如果只是生成了文档、发出了提醒、或系统日期推进了 Day N，**都不能自动视为已完成学习**。

### 2. 提醒进度（系统是否发出提醒）
提醒脚本运行结果只记录在：
- `progress/daily-reminder-history.csv`

它只能说明“提醒是否发送成功/失败”，不能代表学习完成。

## 当前可信状态

- 计划起始日（Day 1）：2026-02-27

- 当前人工确认完成度：**未知 / 待补录**
- 当前提醒记录文件：`progress/daily-reminder-history.csv`

## 状态定义

- `planned`：按计划今天应该学到这里
- `sent`：提醒已成功发出
- `failed_*`：提醒链路失败，但不影响真实学习状态
- `done`：学习者已完成当天学习（需要人工确认）
- `skipped`：明确跳过
- `backfill`：后补学习或后补记录

## 建议更新方式

### 每天学习后
至少补 3 项：
- 完成的是哪一天
- 一条 Go vs Java 差异
- 一个最小可运行示例 / 或说明今天没写出来卡在哪

### 维护时不要混淆
不要把以下事件写成“已完成学习”：
- 发送了提醒
- 生成了 Markdown
- 自动脚本跑到了某一天
- 仅仅更新了 README / 模板 / 文档

## 目前已知情况

### 提醒链路观察
- `daily-push/send.sh` 会按当前日期推算 Day N，并写入 `daily-reminder-history.csv`
- 历史 CSV 里曾出现同一天既有 `failed_no_openclaw` 又有 `sent` 的情况，说明它更像“脚本运行日志”而不是“唯一学习事实”
- 因此后续应把 CSV 视为**提醒事件流水**，不要直接把它当作学习进度表

## 后续建议

1. 如果要补齐真实进度，建议从 Day 1 开始逐天补“done / skipped / backfill”
2. 如果只需要维持提醒系统，保持 `daily-push/` 与 CSV 即可
3. 如果未来要自动汇总进度，建议新增一个独立的、人工确认优先的 `progress/checkpoints.md`，不要复用提醒流水
